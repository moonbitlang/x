# crypto 包代码审查报告

日期：2026-03-12

## 结论

整体上，SHA-1/SHA-224/SHA-256/SM3 的主算法实现从现有测试向量看没有明显算错，主要问题集中在 API 边界和 ChaCha 的误用防护。

## Findings

1. 高：`ChaCha` 会在计数器耗尽后静默回绕，导致同一 `key/nonce` 下复用 keystream。`ChaCha::transform` 每产出一个 64-byte block 就直接 `self.counter += 1`，没有在 `0xffffffff` 后报错或拒绝继续；旧的一次性 `chacha()` 也有同样问题。对 ChaCha20 这类流密码，这是实质性安全风险。  
   参考：[crypto/chacha.mbt](/home/foo/works/x/crypto/chacha.mbt#L468) [crypto/chacha.mbt](/home/foo/works/x/crypto/chacha.mbt#L478) [crypto/chacha.mbt](/home/foo/works/x/crypto/chacha.mbt#L349)

2. 中：`MD5::finalize_into` 的 `offset` 用错方向了，传给了 `src_offset` 而不是 `dst_offset`。结果是 `offset != 0` 时会越界 panic，公开 trait 契约被破坏；只是现在包内调用都用 `offset=0`，所以现有测试没暴露出来。  
   参考：[crypto/md5.mbt](/home/foo/works/x/crypto/md5.mbt#L66) [crypto/md5.mbt](/home/foo/works/x/crypto/md5.mbt#L71) [crypto/traits.mbt](/home/foo/works/x/crypto/traits.mbt#L27)

3. 中：`SHA256::new(reg=...)` 明确支持自定义初始寄存器，但 `reset()` 会无条件重置成标准 SHA-256 IV，导致算法语义被悄悄改掉。任何基于自定义 IV 的用法在调用 `reset()` 后都会变成普通 SHA-256；`hmac()` 又依赖 `reset()`，所以这类 hasher 传进去会直接算错。  
   参考：[crypto/sha256.mbt](/home/foo/works/x/crypto/sha256.mbt#L27) [crypto/sha256.mbt](/home/foo/works/x/crypto/sha256.mbt#L200) [crypto/hmac.mbt](/home/foo/works/x/crypto/hmac.mbt#L38)

4. 中低：仍然公开的 deprecated `chacha8/chacha12/chacha20` API 把 nonce 设计成单个 `UInt`，内部再扩成 `[nonce, nonce, nonce]`。这会把 nonce 空间从 96 bit 压成 32 bit，大幅提高误复用概率，而且与 RFC 8439 的 96-bit nonce 语义不一致。  
   参考：[crypto/chacha.mbt](/home/foo/works/x/crypto/chacha.mbt#L280) [crypto/chacha.mbt](/home/foo/works/x/crypto/chacha.mbt#L284) [crypto/chacha.mbt](/home/foo/works/x/crypto/chacha.mbt#L289)

## 补充说明

- deprecated 但仍导出的 API 也纳入了审查范围；如果确认外部不会再调用，第 4 条可降级。
- 现有 `moon test crypto` 为 `31/31` 通过，但没有覆盖上述边界条件。
- `MD5` 和 `SHA1` 仅适合兼容性场景，不适合新的安全用途。
